## Issue 1 — Critical: new students cannot start the diagnostic after planning

**Status:** Confirmed

**Root cause:** Two functions collaborate to create a redirect loop:

1. `web/lib/student/journeyState.ts` L83-89 — the journey state machine has no
   "planning_complete_awaiting_diagnostic" state. A student who finished planning
   but hasn't started any diagnostic falls into `planning_required` because
   `hasDiagnosticSnapshot` is false and `hasInProgressDiagnostic` is false.

2. `web/lib/student/journeyRouting.ts` L94 — `resolveDiagnosticEntryRoute()`
   checks `journeyState === "planning_required"` first and immediately returns
   the planning route, **before** ever inspecting `hasPlanningProfile`.

3. `web/app/diagnostico/page.tsx` L37-39 — the server component calls
   `resolveDiagnosticEntryRoute(journey)` and redirects whenever the result
   differs from `/diagnostico`. So the student loops back to planning.

**Fix:**

In `resolveDiagnosticEntryRoute`, when the state is `planning_required` the
function must check `hasPlanningProfile` before redirecting:

```typescript
// web/lib/student/journeyRouting.ts — resolveDiagnosticEntryRoute
if (journeySnapshot.journeyState === "planning_required") {
  if (journeySnapshot.hasPlanningProfile) {
    return DIAGNOSTIC_ROUTE;  // planning done → allow diagnostic entry
  }
  return PLANNING_ROUTE;
}
```

Additionally, the contract test that encodes the current (broken) behavior must
be updated:

```typescript
// web/scripts/verifyPortalJourneyContract.ts L121-127
assert(
  resolveDiagnosticEntryRoute({
    journeyState: "planning_required",
    hasPlanningProfile: true,
  }) === "/diagnostico",
  "Planning-complete students must be allowed into diagnostic entry"
);
```

**Files to change:**
- `web/lib/student/journeyRouting.ts` — `resolveDiagnosticEntryRoute()`
- `web/scripts/verifyPortalJourneyContract.ts` — update assertion

## Issue 2 — High: diagnostic resume creates duplicate attempts and wipes progress

**Status:** Confirmed

**Root cause:** Two-part problem on the client side plus missing guard on the server.

1. `web/app/diagnostico/hooks/useDiagnosticFlow.bootstrap.ts` L80 — the
   bootstrap effect unconditionally calls `startTest()` on every mount (including
   page refresh).

2. `web/app/diagnostico/hooks/useDiagnosticFlow.actions.ts` L94-95 — `startTest`
   begins by calling `clearAllDiagnosticData()` (wipes localStorage) then calls
   `startTestAttempt()` which hits `POST /api/diagnostic/start`.

3. `web/app/api/diagnostic/start/route.ts` L18-25 — the server endpoint blindly
   inserts a new `testAttempts` row every time. No check for existing in-progress
   attempts (unlike the full-test start endpoint, which does check).

**Fix (two layers):**

### A. Server: deduplicate in `/api/diagnostic/start`

Before inserting, check for an in-progress diagnostic attempt for the user.
If one exists, return it instead of creating a new one:

```typescript
// web/app/api/diagnostic/start/route.ts
const [existing] = await db
  .select({ id: testAttempts.id })
  .from(testAttempts)
  .where(
    and(
      eq(testAttempts.userId, userId),
      isNull(testAttempts.completedAt),
      isNull(testAttempts.testId) // diagnostics have no testId
    )
  )
  .orderBy(desc(testAttempts.startedAt))
  .limit(1);

if (existing) {
  return NextResponse.json({
    success: true,
    attemptId: existing.id,
    resumed: true,
  });
}
// ...then the existing insert logic
```

### B. Client: resume instead of restart

The bootstrap should check for an existing attempt (via `getStoredAttemptId()`
from localStorage) and skip `clearAllDiagnosticData()` + `startTest()` when one
exists, instead restoring state from localStorage:

```typescript
// web/app/diagnostico/hooks/useDiagnosticFlow.bootstrap.ts
// Inside bootstrapStudentSession(), before calling startTest():
const savedAttemptId = getStoredAttemptId();
const savedResponses = getStoredResponses();

if (savedAttemptId && savedResponses.length > 0) {
  // Resume: restore state from localStorage
  params.setAttemptId(savedAttemptId);
  // Restore question index, stage, etc. from saved responses
  // Then set screen to "question"
  return;
}

// No saved state → start fresh
await params.startTest();
```

**Files to change:**
- `web/app/api/diagnostic/start/route.ts` — add deduplication guard
- `web/app/diagnostico/hooks/useDiagnosticFlow.bootstrap.ts` — resume logic
- `web/app/diagnostico/hooks/useDiagnosticFlow.actions.ts` — `startTest` should
  not be the default bootstrap path
- `web/lib/diagnostic/storage.ts` — may need a `getStoredAttemptId()` export if
  one doesn't exist

## Issue 3 — Medium: resumed full tests show fractional seconds in the timer

**Status:** Confirmed

**Root cause:** Floating-point minutes flow through to the display layer.

1. `web/app/api/student/full-test/start/route.ts` L33-35 — calculates
   `remainingMinutes` as `totalMinutes - (elapsed / 60_000)`, which is a float
   like `148.7853...`.

2. `web/app/portal/test/useFullTestTimer.ts` L47 — initializes state as
   `timeLimitMinutes * 60`, producing a float like `8927.118...`.

3. `useFullTestTimer.ts` L77 — the interval does `prev - 1` each tick, which
   preserves the fractional part forever.

4. `formatTime()` L100 — `totalSeconds % 60` produces `47.118...`, and
   `String(47.118)` renders the decimals.

**Fix (one line, server side):**

Floor the seconds on initialization. The cleanest fix is on the server:

```typescript
// web/app/api/student/full-test/start/route.ts L35
const remainingMinutes = Math.max(0, Math.floor((totalMinutes * 60 - elapsed / 1000)) / 60);
```

Or more simply, floor the total seconds on the client:

```typescript
// web/app/portal/test/useFullTestTimer.ts L47
const [timeRemaining, setTimeRemaining] = useState(
  Math.floor(timeLimitMinutes * 60)
);
```

And also in the effect that syncs the value (L60):

```typescript
useEffect(() => {
  setTimeRemaining(Math.floor(timeLimitMinutes * 60));
}, [timeLimitMinutes]);
```

**Recommended approach:** Apply the `Math.floor()` on the client side (both
L47 and L60) so any future float input is always safe. This is a 2-character
change per line.

**Files to change:**
- `web/app/portal/test/useFullTestTimer.ts` — `Math.floor()` on initialization

## Issue 4 — Medium: profile reminder toggles persist but load with stale UI state

**Status:** Confirmed

**Root cause:** The profile page never fetches the current reminder preferences
on mount. It only fetches `/api/student/me` (which doesn't include reminders).

1. `web/app/portal/profile/page.tsx` L42-62 — `useProfileData()` sets
   `reminders` to `null` initially, then only fetches `/api/student/me`.

2. L130-131 — `NotificationsSection` defaults both toggles to `true` when
   `reminders` is `null`: `reminders?.reminderInApp ?? true`.

3. No GET endpoint exists for `/api/student/reminders/preferences`. The POST
   endpoint saves correctly but the data is never loaded back.

**Fix (two options, pick one):**

### Option A: Add a GET handler to the reminders endpoint

Add a `GET` export to
`web/app/api/student/reminders/preferences/route.ts` that reads from
`studentPlanningProfiles` and returns `{ reminderInApp, reminderEmail }`.
Then call it from `useProfileData()` on mount.

### Option B: Include reminders in `/api/student/me` response

Expand the `/api/student/me` query to join `studentPlanningProfiles` and return
`reminderInApp` / `reminderEmail` alongside the profile data. Then initialize
`reminders` state from that response.

**Recommended approach:** Option A (separate endpoint) keeps the `/me` response
focused and avoids coupling.

Then in the profile page, add a fetch on mount:

```typescript
// Inside useProfileData() load function, after fetching /api/student/me:
const remRes = await fetch("/api/student/reminders/preferences", {
  credentials: "include",
});
const remData = await remRes.json();
if (remData.success) {
  setReminders(remData.data);
}
```

**Files to change:**
- `web/app/api/student/reminders/preferences/route.ts` — add `GET` handler
- `web/app/portal/profile/page.tsx` — fetch reminders on mount

## Issue 5 — Low: weekly mission copy pluralizes "sesión" incorrectly

**Status:** Confirmed

**Root cause:** Spanish accentuation rule — the accent on "sesión" must drop when
adding the plural suffix "-es". The current template concatenates
`"sesión" + "es"` → `"sesiónes"` (wrong). Correct plural: `"sesiones"`.

`web/app/portal/DashboardSections.tsx` L258:
```typescript
`${remaining} sesión${remaining !== 1 ? "es" : ""} más para completar`
```

**Fix:**

Use a ternary that selects the full word:

```typescript
`${remaining} ${remaining !== 1 ? "sesiones" : "sesión"} más para completar`
```

**Files to change:**
- `web/app/portal/DashboardSections.tsx` — L258

## Issue 6 — Low: projection helper note shown after latest result is already a full test

**Status:** Confirmed

**Root cause:** `web/app/portal/progress/ProgressCharts.tsx` L304-307 renders
the note `"Un test completo puede subir el techo"` unconditionally inside
`ProjectionSection`. There's no check for whether the latest score already comes
from a full test.

**Fix:**

The `scoreHistory` array in `ProgressData` already carries a `type` field
(`"short_diagnostic" | "full_test"`) on each data point. Pass the latest type
down to `ProjectionSection` and conditionally render:

```typescript
// ProgressCharts.tsx — ProjectionSection props
latestTestType?: "short_diagnostic" | "full_test" | null;

// Inside the component, replace the unconditional note:
{latestTestType !== "full_test" ? (
  <p className="text-xs text-gray-400">
    Proyección limitada por tu último test. Un test completo puede subir
    el techo.
  </p>
) : null}
```

The parent (`ProgressClient.tsx`) derives the value:

```typescript
const latestType = data.scoreHistory.length > 0
  ? data.scoreHistory[data.scoreHistory.length - 1].type
  : null;
```

**Files to change:**
- `web/app/portal/progress/ProgressCharts.tsx` — add prop, conditional render
- `web/app/portal/progress/ProgressClient.tsx` — pass `latestTestType` prop
